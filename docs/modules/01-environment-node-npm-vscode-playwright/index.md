# Модуль 01: Окружение, Node.js, npm, VS Code и Playwright

## Цель модуля

После этого модуля вы сможете подготовить базовое рабочее окружение Automation QA: установить и проверить Node.js и npm, создать первый проект на Playwright, запустить тесты, открыть HTML report и добавить удобные npm-скрипты для ежедневной работы.

Также вы поймете, зачем Automation QA нужны инструменты вокруг тестов: зависимости, структура проекта, `package.json`, `node_modules`, терминал, VS Code и test runner.

## Почему это важно для Automation QA

Automation QA — это не только написание тестов. В реальной работе нужно уверенно пользоваться инструментами, понимать структуру проекта, устанавливать зависимости, запускать команды, читать ошибки, настраивать скрипты и анализировать отчёты.

Если окружение настроено неправильно, тесты могут не запускаться даже при корректном коде. Поэтому первый шаг в автоматизации — понять, из каких частей состоит рабочая среда и как они связаны между собой.

## 1. Что такое рабочее окружение Automation QA

Рабочее окружение Automation QA — это набор инструментов, которые нужны, чтобы писать, запускать и поддерживать автоматизированные тесты.

Обычно окружение включает:

- Node.js для запуска JavaScript и инструментов TypeScript вне браузера.
- npm для установки пакетов и запуска скриптов.
- VS Code как редактор кода.
- Playwright как фреймворк и test runner.
- Браузеры, которые Playwright использует для запуска тестов.
- Git и GitHub для хранения кода и совместной работы.
- Терминал для выполнения команд.

Для Manual QA это похоже на набор инструментов для тестирования: браузер, DevTools, система управления тестами, баг-трекер и тестовая документация. В Automation QA добавляются редактор кода, зависимости, командная строка и отчёты.

## 2. Node.js

Node.js — это среда выполнения JavaScript вне браузера. Обычно JavaScript выполняется внутри браузера, например на странице сайта. Node.js позволяет запускать JavaScript и инструменты на вашем компьютере через терминал.

TypeScript сам по себе тоже не запускается напрямую в браузере или терминале как обычный готовый код. В проекте используются инструменты, которые компилируют или обрабатывают TypeScript. Эти инструменты работают через Node.js.

Playwright для TypeScript использует Node.js, потому что:

- test runner Playwright запускается в Node.js.
- npm-пакеты устанавливаются и управляются через экосистему Node.js.
- команды Playwright выполняются из терминала.
- тесты, конфигурация и отчёты обрабатываются инструментами Node.js.

Для обучения и реальной работы рекомендуется LTS-версия Node.js. LTS означает Long Term Support: такая версия стабильнее, дольше поддерживается и обычно лучше подходит для production и командной разработки.

Проверить установку Node.js и npm можно командами:

```bash
node -v
npm -v
```

Если обе команды показывают номера версий, значит Node.js и npm доступны в терминале.

## 3. npm

npm — это менеджер пакетов для Node.js. Он помогает устанавливать пакеты, управлять зависимостями и запускать скрипты из `package.json`.

Пакет — это готовая библиотека или инструмент. Например, Playwright устанавливается как npm-пакет.

Зависимость — это пакет, который нужен проекту для работы, разработки или тестирования. Зависимости бывают двух основных типов:

- `dependencies` — зависимости, которые нужны приложению во время работы.
- `devDependencies` — зависимости, которые нужны для разработки, тестирования, сборки или проверки кода.

В проекте автоматизации Playwright обычно находится в `devDependencies`, потому что он нужен для разработки и запуска тестов, а не для работы пользовательского приложения.

### package.json

`package.json` — главный файл проекта Node.js. В нем описаны название проекта, скрипты, `dependencies` и `devDependencies`.

Пример поля `scripts` в `package.json`:

```json
{
  "scripts": {
    "test": "playwright test"
  }
}
```

После этого команду можно запускать так:

```bash
npm run test
```

### package-lock.json

`package-lock.json` фиксирует точные версии установленных пакетов и их внутренних зависимостей. Это помогает всей команде устанавливать одинаковые версии и получать более предсказуемый результат.

Обычно `package-lock.json` нужно добавлять в commit в репозитории.

### node_modules

`node_modules` — это папка, куда npm устанавливает пакеты. Она может быть очень большой, потому что содержит все зависимости проекта.

`node_modules` не нужно добавлять в commit в Git. Вместо этого в репозитории хранят `package.json` и `package-lock.json`, а каждый разработчик или CI-сервер восстанавливает зависимости командой `npm install` или `npm ci`.

### npm install

`npm install` устанавливает зависимости из `package.json` и обновляет `package-lock.json`, если это необходимо.

```bash
npm install
```

Эта команда часто используется локально во время разработки.

### npm ci

`npm ci` устанавливает зависимости строго по `package-lock.json`. Команда обычно используется в CI, потому что дает более чистую и воспроизводимую установку.

```bash
npm ci
```

### npm run

`npm run` запускает скрипт из `package.json`.

```bash
npm run test
```

Это удобно, потому что команда становится короткой и одинаковой для всей команды.

### npx

`npx` запускает команду пакета без необходимости писать полный путь до исполняемого файла в `node_modules`.

Например:

```bash
npx playwright test
```

Эта команда запускает локально установленный Playwright test runner.

## 4. VS Code

VS Code — это редактор кода, в котором удобно писать, запускать и анализировать автоматизированные тесты.

Для Automation QA VS Code полезен, потому что в нем есть:

- проводник файлов для просмотра структуры проекта;
- встроенный терминал для запуска команд;
- поиск по файлам и проекту;
- подсветка синтаксиса для TypeScript;
- подсказки и автодополнение;
- расширения для Playwright, линтинга, форматирования и Git;
- удобное чтение ошибок и переход к нужной строке кода.

Automation QA часто работает в цикле: изменить тест, запустить команду, прочитать ошибку, исправить код, снова запустить тест. VS Code помогает делать это в одном месте.

Рекомендуемые расширения:

- Playwright Test for VS Code — запуск и отладка тестов Playwright из редактора.
- ESLint — подсветка проблем с качеством кода.
- Prettier — автоматическое форматирование кода.
- GitLens — расширенная работа с историей Git и авторством изменений.
- DotENV — подсветка файлов `.env`, которые часто используются для переменных окружения.

## 5. Установка Playwright

Playwright можно установить двумя способами: через терминал или через расширение VS Code.

### Вариант 1. Установка через терминал

В новом учебном проекте выполните команду:

```bash
npm init playwright@latest
```

Эта команда создаст проект Playwright и предложит несколько вариантов настройки.

Типичные ответы для обучения:

- TypeScript — yes, потому что курс использует TypeScript.
- папка для тестов — можно оставить стандартную папку `tests`.
- GitHub Actions — yes, если хотите сразу увидеть пример настройки CI; no, если пока хотите сосредоточиться только на локальном запуске.
- установка браузеров — yes, чтобы Playwright сразу скачал браузеры для тестов.

### Вариант 2. Установка через расширение VS Code

Расширение Playwright Test for VS Code умеет создавать новый проект Playwright через интерфейс. Это удобно, если вы только привыкаете к терминалу.

Но Automation QA все равно должен понимать команды терминала, потому что они используются в CI, документации, онбординге и командной работе.

Важно: в этом репозитории мы не создаем учебный проект автоматизации. Проект Playwright нужно создавать отдельно, например в отдельной папке на вашем компьютере.

## 6. Структура проекта после установки

После установки проект Playwright обычно содержит такие файлы и папки:

```text
tests/
playwright.config.ts
package.json
package-lock.json
node_modules/
```

Что это значит:

- `tests/` — папка с файлами тестов.
- `playwright.config.ts` — файл конфигурации Playwright: браузеры, повторные запуски, отчёты, `baseURL` и другие настройки.
- `package.json` — скрипты и зависимости проекта.
- `package-lock.json` — зафиксированные точные версии зависимостей.
- `node_modules/` — установленные пакеты, которые не нужно добавлять в commit в Git.

## 7. Первый запуск тестов

Основные команды для первого запуска:

```bash
npx playwright test
npx playwright test --headed
npx playwright test --ui
npx playwright show-report
```

Что делает каждая команда:

- `npx playwright test` запускает все тесты в headless-режиме, то есть браузеры открываются без видимого окна.
- `npx playwright test --headed` запускает тесты с видимым окном браузера. Это полезно для обучения и отладки.
- `npx playwright test --ui` открывает режим Playwright UI, где можно выбирать тесты, смотреть шаги и быстрее разбираться в поведении.
- `npx playwright show-report` открывает HTML report после запуска тестов.

HTML report помогает увидеть, какие тесты прошли, какие упали, сколько времени занял запуск и где произошла ошибка.

## 8. Объяснение первого теста

Пример первого теста:

```ts
import { test, expect } from '@playwright/test';

test('Playwright homepage has correct title', async ({ page }) => {
  await page.goto('https://playwright.dev/');

  await expect(page).toHaveTitle(/Playwright/);
});
```

Разберем по строкам:

- `import { test, expect } from '@playwright/test';` подключает из Playwright две главные функции: `test` для описания тестового случая и `expect` для проверки результата.
- `test('Playwright homepage has correct title', ...)` создает тест с понятным названием.
- `async` показывает, что внутри теста будут асинхронные операции: действия в браузере, переходы по страницам и проверки.
- `({ page })` — это Playwright fixture. `page` представляет вкладку браузера, с которой взаимодействует тест.
- `await page.goto('https://playwright.dev/');` открывает сайт Playwright.
- `expect(page).toHaveTitle(/Playwright/)` проверяет, что заголовок страницы содержит слово `Playwright`.
- Такая проверка называется assertion: она сравнивает ожидаемый результат с фактическим.
- `await` говорит test runner подождать завершения действия или проверки перед переходом к следующей строке.

Пока не нужно глубоко разбирать `async` и `await`. В этом модуле важно понять общий смысл: Playwright выполняет действия в браузере не мгновенно, поэтому тест должен дожидаться результата. Подробно асинхронный код будет разобран в отдельном модуле.

## 9. npm-скрипты

В `package.json` можно добавить скрипты для частых команд:

```json
{
  "scripts": {
    "test": "playwright test",
    "test:headed": "playwright test --headed",
    "test:ui": "playwright test --ui",
    "report": "playwright show-report"
  }
}
```

После этого команды будут выглядеть так:

```bash
npm run test
npm run test:headed
npm run test:ui
npm run report
```

Скрипты полезны, потому что:

- делают команды короче;
- стандартизируют запуск тестов для всей команды;
- уменьшают риск опечаток;
- помогают новичкам быстрее понять, как работать с проектом;
- используются в CI-пайплайнах.
